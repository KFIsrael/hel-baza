Конкретные фитинги с размерами и артикулами. Цены загружаются из 1С через fitting_prices.

ПолеТипОписание

fitting_prices — Цены фитингов

Мультивалютные цены с типами (опт/розница/маркетплейсы). Автозагрузка из 1С.

Иерархия: vehicle_brands → vehicle_models → vehicle_modifications

Трёхуровневая структура: Бренд (Toyota) → Модель (Camry XV70) → Модификация (2018–2024, 2.5L, Sedan). Модификация привязывается к OEM через таблицу applicability.

⚠️ Марка/модель/год — информационные поля, НЕ связующие. Связь аналогов идёт через OEM кроссы.

profiles — Профили пользователей

Расширение auth.users Supabase. Роль определяет доступ: admin, manager, dealer, client.

balance_transactions — Баланс дилера

Пополнение и списания. Каждый поисковый запрос дилера списывает фиксированную сумму.

search_logs — Лог запросов

Все поисковые запросы с аналитикой: тип, текст, кол-во результатов, IP, user-agent. Для статистики «что ищут».

export_channels — Каналы продаж

Минимум 6 каналов: сайт, Ozon, Wildberries, Emex, ZZAP, CRM. Каждый канал имеет свой маппинг полей и тип цен.

export_templates — Шаблоны выгрузки

Настраиваемые шаблоны Excel: какие поля выгружать, какие фильтры применять. Менеджеры могут создавать свои шаблоны.

👥 3. Роли и права доступа

🔴 Админ

Полный доступ ко всему: управление базой, роли, пользователи, аналоги, схемы, OEM, каналы экспорта, настройки цен, логи. Row Level Security (RLS) — без ограничений.

🟠 Менеджер

PIM: создание/редактирование OEM, схем, фитингов, загрузка изображений, связывание аналогов, конструктор шланга с сохранением в базу, Excel импорт/экспорт.

🔵 Дилер

Поиск (OEM, VIN, марка/модель). Баланс с пополнением, списание за каждый запрос. Просмотр карточек, цены дилерские. История запросов, статистика.

🟢 Клиент

Поиск (бесплатно или с ограничением). Конструктор кастомного шланга. Просмотр карточек, розничные цены. Заказ фитингов → 1С. Личный кабинет, история.

🔧 4. Конструктор шланга

Режим «Клиент» — Кастомный шланг

1
Выбор левого фитинга: тип → размер → угол
2
Выбор правого фитинга: тип → размер → угол
3
Выбор цвета шланга (визуальные свотчи)
4
Длина шланга (15–200+ см, шаг 1 см)
5
Опции: Stealth Heatshrink, Support (1-3), назначение (Brake/Clutch)
6
Превью: изображение собранного шланга + расчёт цены
7
Аксессуары (банджо болты и т.п.)
8
Заказ → оплата → отправка в 1С/Bitrix

Режим «Менеджер» — Добавление в базу

1
Всё то же + ввод OEM номера
2
Система проверяет: есть ли такой OEM в базе?
3
Если есть → показать карточку, предложить обновить
4
Если нет → «Подгрузить кроссы»: поиск аналогов в базе кроссировок
5
Если кроссов нет → «Заполнить вручную» + статус «new_oem»
6
Добавить применимость (марка/модель/год)
7
Загрузить изображение/схему
8
Генерация SKU → сохранение схемы как ИД в базу

Формула расчёта цены

price = (fitting_left.price + fitting_right.price) + (hose_length_cm × price_per_cm) + heatshrink_price + Σ(supports.price) + Σ(accessories.price)

Итоговая цена умножается на коэффициент price_type (retail ×1.0, wholesale ×0.7, ozon ×1.15 и т.д.)

Типы поиска

ТипОписаниеТаблицы
OEMПрямой поиск по номеру OEMoem_numbers → oem_scheme_links → schemes
АналогПоиск через кроссыoem_cross_references → oem_numbers → schemes
Марка/Модель/ГодФильтр по применимостиvehicle_* → applicability → oem_numbers → schemes
VINРасшифровка VIN → марка/модель/год → поискVIN decode API → applicability chain

Оптимизация поиска

• GIN индексы на oem_number (pg_trgm) для нечёткого поиска

• Full-text search по описаниям (tsvector)

• Materialized view для «плоской» карточки (денормализация для скорости)

• Нормализация OEM: удаление пробелов, дефисов, приведение к верхнему регистру перед поиском

Для дилеров: каждый поисковый запрос → списание с баланса + запись в search_logs

🔗 9. Интеграции

1С → Supabase

Входящие: Цены фитингов, наличие, спецификации

Исходящие: Заказы на фитинги, новые схемы

Механизм: REST API (Edge Function) с API-ключом. 1С отправляет JSON с ценами → Edge Function валидирует и обновляет fitting_prices. Поддержка ручного и автоматического режима (по расписанию).

Bitrix24 CRM

Исходящие: Заказы, заявки с лендинга

Входящие: Статусы заказов

Механизм: Bitrix24 REST API. При создании заказа → webhook отправляет данные в CRM как сделку.

Маркетплейсы (Drive, Emex, ZZAP)

Исходящие: Каталог товаров по шаблону

Механизм: Генерация Excel/CSV по шаблону export_template. Ручная выгрузка (v1), API-push (v2).

VIN Decode

Входящие: Расшифровка VIN → марка/модель/год

Механизм: Внешний API (NHTSA бесплатный или коммерческий). Edge Function как прокси с кэшированием результатов.

🛡️ 10. Безопасность и анти-скрейп

Анти-парсинг

• Rate limiting — лимит запросов по IP + по user (настраиваемый)

• API токены — все запросы через авторизацию (JWT Supabase)

• Watermark на схемы — наложение при отдаче изображений (Edge Function + Sharp)

• SSR — Server-side rendering, данные не в открытом JSON

• Honeypot поля — ловушки для ботов

• CAPTCHA — при превышении лимита запросов

Защита данных

• RLS (Row Level Security) — на каждой таблице

• Supabase Auth — JWT, email/password, magic link

• Роли в БД — через profiles.role + RLS policies

• Аудит-лог — все действия с критичными данными

• Бэкап — автоматический через Supabase

• CORS — только разрешённые домены
